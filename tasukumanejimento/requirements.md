# 要件定義書：タスク管理ボード

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | タスク管理ボード |
| 作成日 | 2026年6月27日 |
| 最終更新日 | 2026年6月29日 |
| 作成者 | マキ |
| バージョン | 2.0 |

### 改訂履歴

| バージョン | 日付 | 変更内容 |
|-----------|------|---------|
| 1.0 | 2026-06-27 | 初版作成（localStorage構成） |
| 2.0 | 2026-06-29 | バックエンド・DB構成に変更（Python/FastAPI/SQLite）、設計ドキュメントを統合 |

### 背景・目的

個人のタスクを「やること・進行中・完了」の3段階で視覚的に管理するため、  
Trello風のカンバンボードをWebアプリとして開発する。  
バックエンドAPIとデータベースを導入し、データの永続化とAPI設計の学習を目的とする。

---

## 2. 対象ユーザー

- **利用者**：開発者本人（シングルユーザー）
- **利用環境**：PCブラウザ（Chrome / Edge / Firefox）
- **利用シーン**：日々のタスクや学習の進捗管理

> **スコープ外**：マルチユーザー対応・認証・ログイン機能は含まない

---

## 3. 用語定義

| 用語 | 定義 |
|------|------|
| タスク | 管理対象の作業単位。タイトル・説明文・期限・ステータスを持つ |
| カード | タスクをUI上で表示する矩形のコンポーネント |
| ステータス | タスクの進捗状態。`todo`・`in_progress`・`done` の3種類 |
| 列 | ステータスごとにカードを縦に並べた表示エリア |

---

## 4. 機能要件

> 機能要件とは「このシステムが何をできなければならないか」を定めたもの。

### 4-1. タスクの追加

| # | 要件 |
|---|------|
| F-01 | タイトル・説明文・期限を入力してタスクを追加できる |
| F-02 | タイトルは必須入力とする。空欄の場合はエラーメッセージを表示する |
| F-03 | 説明文・期限は任意入力とする |
| F-04 | 追加ボタンのクリック、またはEnterキーでタスクを追加できる |
| F-05 | 追加後、入力欄はリセットされる |

### 4-2. タスクの表示

| # | 要件 |
|---|------|
| F-06 | タスクはカード形式で表示する |
| F-07 | カードにはタイトル・説明文・期限を表示する |
| F-08 | 説明文・期限が未入力の場合、その項目はカードに表示しない |
| F-09 | 期限が本日より前の場合、期限を赤字・「⚠️ 期限切れ」と表示する |

### 4-3. タスクの移動

| # | 要件 |
|---|------|
| F-10 | タスクを「やること → 進行中 → 完了」の順に進めることができる |
| F-11 | タスクを前の列に戻すこともできる |
| F-12 | 一番左の列（やること）では「戻す」ボタンを表示しない |
| F-13 | 一番右の列（完了）では「進める」ボタンを表示しない |

### 4-4. タスクの削除

| # | 要件 |
|---|------|
| F-14 | 各カードに削除ボタンを設ける |
| F-15 | 削除前に確認ダイアログを表示する |

### 4-5. データの保存

| # | 要件 |
|---|------|
| F-16 | タスクの追加・移動・削除はバックエンドAPIを通じてDBに保存する |
| F-17 | ページを開いたとき、APIからタスク一覧を取得して表示する |

### 7-3. コンポーネント詳細

#### A. ヘッダー

| 項目 | 仕様 |
|------|------|
| テキスト | 「タスク管理ボード」固定 |
| 配置 | 画面上部・中央揃え |

#### B. 入力フォーム

| 項目 | 要素 | 仕様 |
|------|------|------|
| タイトル | `<input type="text">` | 必須・最大100文字・プレースホルダー「タスクのタイトル」 |
| 説明文 | `<textarea>` | 任意・3行表示・プレースホルダー「説明（任意）」 |
| 期限 | `<input type="date">` | 任意・YYYY-MM-DD形式 |
| 追加ボタン | `<button>` | テキスト「タスクを追加」・クリックまたはEnterキーで送信 |

**バリデーション**

| 条件 | 表示内容 |
|------|---------|
| タイトルが空欄で追加ボタン押下 | タイトル入力欄の下に「タイトルを入力してください」を赤字表示 |
| 追加成功後 | 全入力欄をリセット・エラーメッセージを消去 |

#### C. カンバンボード（列）

| 列名 | ステータス値 | アイコン |
|------|------------|---------|
| やること | `todo` | 📋 |
| 進行中 | `in_progress` | 🔄 |
| 完了 | `done` | ✅ |

各列にはタスク件数を `（N件）` 形式で表示する。

#### D. タスクカード

```
┌───────────────────────────┐
│ タスクのタイトル           │  ← タイトル（太字）
│                           │
│ 説明文テキストが入ります。  │  ← 説明文（任意表示）
│                           │
│ 📅 2026-07-10             │  ← 期限（任意表示）
│                           │
│        [←戻す] [進む→] [🗑]│  ← アクションボタン
└───────────────────────────┘
```

**カード表示仕様**

| 項目 | 仕様 |
|------|------|
| タイトル | 常に表示・太字 |
| 説明文 | 入力がある場合のみ表示 |
| 期限 | 入力がある場合のみ表示 |
| 期限（通常） | 「📅 YYYY-MM-DD」形式で表示 |
| 期限（期限切れ） | 「⚠️ 期限切れ YYYY-MM-DD」を赤字で表示（本日より前の日付） |
| 期限（本日） | 通常表示（期限切れとしない） |

**アクションボタン表示ルール**

| ステータス | 戻すボタン | 進むボタン | 削除ボタン |
|-----------|-----------|-----------|-----------|
| `todo`（やること） | 非表示 | 表示 | 表示 |
| `in_progress`（進行中） | 表示 | 表示 | 表示 |
| `done`（完了） | 表示 | 非表示 | 表示 |

**削除確認ダイアログ**

| 項目 | 仕様 |
|------|------|
| 表示タイミング | 削除ボタン押下時 |
| メッセージ | 「このタスクを削除しますか？」 |
| ボタン | 「OK」で削除実行 / 「キャンセル」で閉じる |

### 7-4. 画面状態一覧

| 状態 | 説明 | 表示内容 |
|------|------|---------|
| 初期表示 | ページ読み込み直後 | APIからタスク一覧を取得して各列に表示 |
| タスクなし | 各列にタスクが0件 | 各列に「タスクはありません」のメッセージを表示 |
| 読み込み中 | API通信中 | （今回は即時応答のため特別な表示なし） |
| バリデーションエラー | タイトル未入力で追加 | エラーメッセージを赤字で表示 |

## 8. 状態遷移

### 8-1. 遷移図

```
                    ┌─────────────┐
                    │   （開始）   │
                    │  タスク追加  │
                    └──────┬──────┘
                           │
                           ▼
                    ┌─────────────┐
              ┌────▶│    todo     │
              │     │  やること    │
              │     └──────┬──────┘
              │            │ 進む→
              │            ▼
              │     ┌─────────────┐
              │     │ in_progress │◀──┐
              │     │   進行中    │   │ 戻す←
              │     └──────┬──────┘   │
              │ 戻す←      │ 進む→    │
              └────────────┤          │
                           ▼          │
                    ┌─────────────┐   │
                    │    done     │───┘
                    │    完了     │
                    └──────┬──────┘
                           │ 削除
                           ▼
                    ┌─────────────┐
                    │  （終了）   │
                    │ タスク削除  │
                    └─────────────┘
```

### 8-2. 遷移一覧

| 遷移ID | 遷移前ステータス | 遷移後ステータス | トリガー | API呼び出し |
|--------|----------------|----------------|---------|------------|
| T-01 | −（新規） | `todo` | 追加ボタン押下 | `POST /tasks` |
| T-02 | `todo` | `in_progress` | 「進む→」ボタン押下 | `PATCH /tasks/{id}/status` |
| T-03 | `in_progress` | `done` | 「進む→」ボタン押下 | `PATCH /tasks/{id}/status` |
| T-04 | `in_progress` | `todo` | 「←戻す」ボタン押下 | `PATCH /tasks/{id}/status` |
| T-05 | `done` | `in_progress` | 「←戻す」ボタン押下 | `PATCH /tasks/{id}/status` |
| T-06 | `todo` | −（削除） | 削除ボタン → OK | `DELETE /tasks/{id}` |
| T-07 | `in_progress` | −（削除） | 削除ボタン → OK | `DELETE /tasks/{id}` |
| T-08 | `done` | −（削除） | 削除ボタン → OK | `DELETE /tasks/{id}` |

### 8-3. 遷移制約

| 制約 | 内容 |
|------|------|
| スキップ不可 | `todo` → `done` への直接遷移は不可。必ず `in_progress` を経由する |
| 逆戻りは1段階のみ | `done` → `todo` への直接戻しは不可。`done` → `in_progress` のみ |
| 削除はどのステータスからも可能 | 全ステータスで削除ボタンを表示する |

## 9. データ設計（ER図）

### 9-1. テーブル一覧

今回はシングルユーザー・単一機能のため、テーブルは `tasks` のみ。

### 9-2. tasks テーブル定義

| カラム名 | 型 | 制約 | 説明 |
|---------|-----|------|------|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | タスクの一意識別子 |
| `title` | VARCHAR(100) | NOT NULL | タスクのタイトル |
| `description` | TEXT | NULL許容 | タスクの説明文 |
| `due_date` | DATE | NULL許容 | 期限（YYYY-MM-DD形式） |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT 'todo' | ステータス（下記参照） |
| `created_at` | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 作成日時 |
| `updated_at` | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 最終更新日時 |

**statusカラムの取りうる値**

| 値 | 表示名 | 説明 |
|----|--------|------|
| `todo` | やること | 未着手のタスク |
| `in_progress` | 進行中 | 作業中のタスク |
| `done` | 完了 | 完了したタスク |

### 9-3. ER図

```
┌──────────────────────────────┐
│            tasks             │
├──────────────────────────────┤
│ PK  id          INTEGER      │
│     title       VARCHAR(100) │
```

### 9-4. 制約まとめ

| 制約 | 対象カラム | 内容 |

| DEFAULT | `status` | 未指定時は `todo` |
| DEFAULT | `created_at` | レコード挿入時の日時を自動セット |
| DEFAULT | `updated_at` | レコード挿入時の日時を自動セット（更新時はアプリ側で更新） |
| CHECK（推奨） | `status` | `todo` / `in_progress` / `done` 以外を拒否 |

## 10. API設計

### 10-1. 基本情報

| 項目 | 内容 |
|------|------|
| ベースURL | `http://localhost:8000` |
| データ形式 | JSON |
| 文字コード | UTF-8 |

### 10-2. エンドポイント一覧

| メソッド | パス | 説明 | 対応遷移 |
|---------|------|------|---------|
| GET | `/tasks` | タスク一覧取得 | 画面初期表示 |
| POST | `/tasks` | タスク新規作成 | T-01 |
| PATCH | `/tasks/{id}/status` | ステータス更新 | T-02〜T-05 |
| DELETE | `/tasks/{id}` | タスク削除 | T-06〜T-08 |

### 10-3. エンドポイント詳細

#### GET /tasks

タスク一覧を取得する。リクエストパラメータなし。

**レスポンス（200 OK）**

```json
[
  {
    "id": 1,
    "title": "FastAPIの勉強",
    "description": "uvicornの使い方を調べる",
    "due_date": "2026-07-05",
    "status": "in_progress",
    "created_at": "2026-06-29T10:00:00",
    "updated_at": "2026-06-29T12:00:00"
  }
]
```

タスクが0件の場合は空配列 `[]` を返す。

---

#### POST /tasks

タスクを新規作成する。

**リクエストボディ**

```json
{
  "title": "フロントを実装する",
  "description": "fetch APIでGET/POSTする",
  "due_date": "2026-07-10"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `title` | string | ✅ | タスクのタイトル（最大100文字） |
| `description` | string | ❌ | 説明文（省略時はnull） |
| `due_date` | string | ❌ | 期限（YYYY-MM-DD形式、省略時はnull） |

**レスポンス（201 Created）**：作成されたタスクをそのまま返す。

**エラーレスポンス（422）**：タイトル未入力時。

---

#### PATCH /tasks/{id}/status

タスクのステータスを更新する。

**リクエストボディ**

```json
{ "status": "in_progress" }
```

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `status` | string | ✅ | `todo` / `in_progress` / `done` のいずれか |

**レスポンス（200 OK）**：更新後のタスクを返す。

**エラーレスポンス**：404（ID不存在）、422（status値不正）。

---

#### DELETE /tasks/{id}

タスクを削除する。

**レスポンス（204 No Content）**：ボディなし。

**エラーレスポンス**：404（ID不存在）。

---

### 10-4. 使用するHTTPステータスコード

| コード | 意味 | 使用場面 |
|--------|------|---------|
| 200 OK | 成功 | GET・PATCH |
| 201 Created | 作成成功 | POST |
| 204 No Content | 削除成功 | DELETE |
| 404 Not Found | リソースなし | 存在しないIDへのPATCH・DELETE |
| 422 Unprocessable Entity | バリデーションエラー | 必須項目不足・不正な値 |

### 10-5. フロントエンドとの対応関係

| ユーザー操作 | API呼び出し |
|------------|------------|
| ページ読み込み | `GET /tasks` |
| 「タスクを追加」ボタン押下 | `POST /tasks` |
| 「進む→」ボタン押下 | `PATCH /tasks/{id}/status` |
| 「←戻す」ボタン押下 | `PATCH /tasks/{id}/status` |
| 「削除」→「OK」押下 | `DELETE /tasks/{id}` |

### 11-2. ユースケース詳細

---

#### UC-01：タスクを追加する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | アプリがブラウザで開かれている |
| 事後条件 | 新しいタスクが「やること」列に表示され、DBに保存されている |

**基本フロー（正常系）**

1. ユーザーがタイトル入力欄にタイトルを入力する
2. ユーザーが必要に応じて説明文・期限を入力する
3. ユーザーが「タスクを追加」ボタンを押す（またはEnterキー）
4. システムが入力値をバリデーションする
5. システムが `POST /tasks` をAPIに送信する
6. APIがDBにタスクを保存し、作成されたタスクを返す
7. システムが「やること」列にカードを追加表示する
8. システムが入力欄をリセットする

**代替フロー（異常系）**

| 条件 | システムの動作 |
|------|-------------|
| タイトルが空欄のまま追加ボタンを押した | 「タイトルを入力してください」を赤字表示し、APIは呼び出さない |
| API通信に失敗した | カードは追加されない（エラーハンドリングは将来対応） |

#### UC-02：タスク一覧を確認する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | なし |
| 事後条件 | 保存済みタスクがステータス別の列に表示されている |

**基本フロー（正常系）**

1. ユーザーがブラウザでアプリのURLを開く
2. システムが `GET /tasks` をAPIに送信する
3. APIがDBからタスク一覧を取得して返す
4. システムが各タスクを `status` の値に応じて列に振り分けて表示する

**代替フロー（異常系）**

| 条件 | システムの動作 |
|------|-------------|
| タスクが1件も登録されていない | 各列に「タスクはありません」を表示する |

#### UC-03：タスクを進める

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 「やること」または「進行中」列にタスクが1件以上存在する |
| 事後条件 | タスクが次のステータスの列に移動し、DBが更新されている |

**基本フロー（正常系）**

1. ユーザーがカードの「進む→」ボタンを押す
2. システムが次のステータス値を決定する（`todo`→`in_progress`、`in_progress`→`done`）
3. システムが `PATCH /tasks/{id}/status` をAPIに送信する
4. APIがDBのステータスを更新し、更新後のタスクを返す
5. システムがカードを元の列から削除し、次の列に移動して表示する

**代替フロー（異常系）**

| 条件 | システムの動作 |
|------|-------------|
| 「完了」列のタスクで進むボタンが押された | 進むボタン自体が表示されないため発生しない |

#### UC-04：タスクを戻す

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 「進行中」または「完了」列にタスクが1件以上存在する |
| 事後条件 | タスクが前のステータスの列に移動し、DBが更新されている |

**基本フロー（正常系）**

1. ユーザーがカードの「←戻す」ボタンを押す
2. システムが前のステータス値を決定する（`in_progress`→`todo`、`done`→`in_progress`）
3. システムが `PATCH /tasks/{id}/status` をAPIに送信する
4. APIがDBのステータスを更新し、更新後のタスクを返す
5. システムがカードを元の列から削除し、前の列に移動して表示する

**代替フロー（異常系）**

| 条件 | システムの動作 |
|------|-------------|
| 「やること」列のタスクで戻すボタンが押された | 戻すボタン自体が表示されないため発生しない |

#### UC-05：タスクを削除する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | いずれかの列にタスクが1件以上存在する |
| 事後条件 | タスクが列から消え、DBから削除されている |

**基本フロー（正常系）**

1. ユーザーがカードの「🗑」削除ボタンを押す
2. システムが確認ダイアログ「このタスクを削除しますか？」を表示する
3. ユーザーが「OK」を押す
4. システムが `DELETE /tasks/{id}` をAPIに送信する
5. APIがDBからタスクを削除する
6. システムが列からカードを削除して表示を更新する

**代替フロー（異常系）**

| 条件 | システムの動作 |
|------|-------------|
| 確認ダイアログで「キャンセル」を押した | 削除をキャンセルし、タスクはそのまま残る |
